# 一体机指标智能问答方案

**方案范围**：OEM + Grafana + AI Gateway + 知识库（Oracle 26ai 或企业自有 RAG） + 客户端使用流程  
**示例对象**：
- Memory HardwareCorrupted
- DiskErrorCount / errOtherCount

---

## 1. 方案定位

本方案的目标，不是重新做一套监控平台，也不是去替代 OEM，而是在现有 OEM 监控基础上，增加一个更容易使用的“智能问答入口”。

整个思路可以概括为：

- **OEM** 负责采集监控数据、保存历史趋势、生成告警和事件
- **Grafana** 负责把趋势和细节看清楚，承接后续钻取
- **AI Gateway** 负责理解用户问题、调用 OEM 和知识库、组织答案
- **知识库** 负责补充 SOP、经验、文档解释和处理建议
- **客户端** 负责提供一个简单入口，让终端用户像提问一样使用系统

这套方案借鉴了 Ask EM 的设计理念，但不依赖 OCI，不修改 OEM 本体，也不通过用户名和密码访问 OEM repository 数据库。

---

## 2. 建设背景与设计约束

### 2.1 为什么要做

目前 OEM 已经覆盖数据库、主机、网络、磁盘、一体机等大量监控项，监控能力本身并不弱。真正的问题在于，一线运维在使用这些数据时仍然存在几个痛点：

1. 页面多、入口散，查问题时需要不断切换页面
2. 同一个问题往往要同时看指标、趋势、告警和经验文档
3. 监控数据虽然有，但并没有自然语言层面的“解释”和“下一步建议”
4. 一些场景明明已经有数据和阈值，却还需要资深工程师手工判断优先级

因此，需要一个更接近日常运维习惯的入口：**用户只提问题，系统自动去取数据、组织证据、给出建议**。

### 2.2 设计约束

本方案必须满足以下约束：

1. **OEM 不做源码级定制**  
   OEM 是商业化产品，不适合做深度定制开发。

2. **不能通过用户名/密码方式访问 OEM repository 数据库**  
   因此，方案不能建立在直连 OMR 的基础上。

3. **优先复用 OEM 官方支持能力**  
   包括 OEM 自身的监控、事件、Metric Extension、REST API，以及 Oracle 官方提供的 Grafana 集成。

4. **使用和维护要简单**  
   终端用户不应承担复杂配置成本，运维团队也不应为此再维护一套重平台。

5. **后续要能接自动化**  
   第一阶段可以只做只读问答，但后面要为 Ansible 或其他自动化流程留出扩展接口。

## 3. 示例对象：两个一体机硬件指标

本方案以两个已经具备基础监控能力的一体机指标为例，说明整套设计如何落地。

### 3.1 指标 A：Memory HardwareCorrupted

该指标通过主机脚本读取 `/proc/meminfo` 中的 `HardwareCorrupted` 字段，经 dcli 汇总每台主机的结果，再以 Metric Extension 的方式挂入 OEM。

其特点如下：

- 目标类型：主机
- Metric Extension：`Check_Mem_HardwareCorrupted`
- 数据列：`Memory_HardwareCorrupted`
- 单位：KB
- 阈值：
  - Warning：`> 0`
  - Critical：`> 16`

它适合回答的问题包括：

- 某台主机当前是否存在硬件内存异常
- 最近 24 小时该值是否持续存在
- 当前是否已经达到严重级别
- 是否值得进一步查看硬件日志或一体机相关告警

### 3.2 指标 B：DiskErrorCount / errOtherCount

该指标通过 `cellcli` 读取 physical disk 的 `errOtherCount`，然后按 `hostname:diskname` 形成多行结果，再挂入 OEM 主机目标。

其特点如下：

- 目标类型：主机
- Metric Extension：`Check_Exadata_Disk_Error_Count`
- 关键字列：`hostname:diskname`
- 数据列：`DiskErrorCount`
- 阈值：
  - Warning：`>= 1`
  - Critical：`>= 10`

它适合回答的问题包括：

- 哪几块盘最近开始出现错误计数
- 某块盘的计数是在增长，还是只是历史残留值
- 当前哪块盘最值得优先关注
- 是否已经形成告警，需要进一步升级处理

### 3.3 为什么这两个指标适合做第一阶段试点

因为这两个指标已经具备完整的基础条件：

- 已有采集脚本
- 已在 OEM 中定义为 Metric Extension
- 已有阈值规则
- 已可形成事件和告警

也就是说，数据基础已经有了，缺的不是采集，而是一个更智能、更易用的使用方式。

## 4. 整体方案架构

整套方案建议由 5 个组件构成：

1. OEM
2. Grafana
3. AI Gateway
4. 知识库
5. 客户端

### 4.1 OEM：监控事实来源

OEM 在本方案中是“数据底座”，负责：

- 采集监控数据
- 保存历史趋势
- 承载 Metric Extension
- 生成 incidents / events
- 提供可控的外部访问接口

本方案中，OEM 只做自己最擅长的事情：**采集、保存、告警**。它不负责自然语言交互，也不负责复杂的智能解释。

### 4.2 Grafana：图表与趋势钻取层

Grafana 在本方案中的角色不是“智能大脑”，而是“可视化承接层”。

它主要负责：

- 展示趋势曲线
- 展示多指标联动
- 展示单主机或单磁盘的详细变化
- 承接 AI Gateway 在回答中附上的 dashboard / panel 链接
- 让用户在需要时继续 drill-down

换句话说，AI 先告诉用户“发生了什么、为什么值得关注、下一步看什么”，Grafana 再让用户真正把图看清楚。

### 4.3 AI Gateway：整个方案的编排中枢

AI Gateway 是整套方案的核心。

它主要负责：

1. 接收用户自然语言问题
2. 识别问题类型和对象
3. 调用 OEM 取监控数据和事件
4. 调用知识库取 SOP / 文档 / 经验
5. 把结果组织成可直接使用的答案
6. 为用户附上 Grafana 的 dashboard / panel 跳转链接

建议把 AI Gateway 设计成一个**内部轻量服务**，而不是一个很重的平台。它的定位是“编排中间层”，不是“第二个监控系统”。

### 4.4 知识库：补足文档与经验层

知识库可以是：

- Oracle 26ai
- 企业自有 RAG 平台
- 其他内部文档检索能力

它主要承载：

- Oracle 官方文档
- 一体机运维手册
- 企业内部 SOP
- 历史故障案例
- 常见误报说明
- 升级和处理建议

它的作用，是把“只会报数”升级为“会解释、会建议”。

### 4.5 客户端：终端用户入口

建议第一阶段客户端优先做成 **CLI**，例如：

```bash
askops "x9mdbadm01 这台主机为什么有硬件内存告警"
askops "x9mceladm01 最近有没有磁盘错误上升"
askops "现在哪些一体机硬件风险最值得先处理"
```

后续如果用户量增多，再考虑扩展到内部 Web 页面。

客户端的原则是：

- 用户只负责提问
- 不要求用户理解 OEM REST API
- 不要求用户知道 metric group 或事件对象
- 不要求用户自己维护复杂连接关系

## 5. 终端用户交互流程

### 5.1 总体流程

终端用户的实际体验应该尽量简单：

1. 用户提问
2. AI Gateway 识别问题类型
3. AI Gateway 去 OEM 与知识库取证
4. AI Gateway 返回结构化答案
5. 用户如需深挖，再点击 Grafana 链接查看趋势和细节

### 5.2 典型流程说明

#### 第一步：用户提问

例如：

```bash
askops "x9mceladm01 最近有没有磁盘错误上升"
```

#### 第二步：AI Gateway 识别问题

系统先判断：

- 这是“趋势变化”还是“当前异常解释”
- 对象是 `x9mceladm01`
- 指标重点是 `DiskErrorCount`
- 时间范围默认看最近 24 小时，或按用户指定范围处理

#### 第三步：系统去 OEM 和知识库取证

AI Gateway 通过 OEM 官方接口去读取：

- 目标信息
- 指标最新值
- 时间序列趋势
- incidents / events

同时去知识库查：

- 该指标一般意味着什么
- 应优先排查什么
- 是否有企业内部标准处理流程

#### 第四步：系统组织答案

返回内容建议固定成：

- 结论
- 证据
- 影响范围
- 下一步建议
- Grafana 跳转链接

#### 第五步：用户点击 Grafana 继续钻取

如果用户希望进一步看图，例如确认是哪块盘、何时开始变化、与其他指标是否联动，就点击链接直接跳到 Grafana 对应 dashboard 或 panel。

## 6. 两个典型问答示例

### 6.1 示例一：内存 HardwareCorrupted

#### 用户提问

```bash
askops "x9mdbadm01 这台主机为什么有硬件内存告警"
```

#### 系统内部处理

1. 识别为“单目标异常解释”
2. 查询 OEM 中该主机的 `Memory_HardwareCorrupted` 最新值
3. 查询最近 24 小时趋势
4. 查询是否已有对应 incident / event
5. 查询知识库中关于该类问题的说明和排查建议
6. 组织回答，并附上主机硬件相关 Grafana 链接

#### 返回示例

- 当前 `Memory_HardwareCorrupted = 32 KB`
- 已超过 Critical 阈值 `16 KB`
- 最近 24 小时从 0 变为 32，并持续存在
- 当前已有 1 条相关硬件类告警
- 建议先确认是否为持续异常，再查看硬件日志和一体机相关告警

### 6.2 示例二：磁盘 DiskErrorCount

#### 用户提问

```bash
askops "x9mceladm01 最近有没有磁盘错误上升"
```

#### 系统内部处理

1. 识别为“趋势查询 + 风险判断”
2. 查询该主机所有物理盘的 `DiskErrorCount`
3. 比较最近一段时间的变化，判断哪些盘开始上升
4. 查询是否已有相关 incidents / events
5. 从知识库补充排查思路
6. 附上 Exadata 磁盘趋势 Grafana 链接

#### 返回示例

- 当前有 2 块盘出现错误计数上升
- 其中 `252:7` 从 0 增至 3
- 已达到 Warning 阈值，但尚未达到 Critical
- 建议优先看该盘是否持续增长，并结合 cell 硬件告警一起判断

## 7. 各组件用途及技术说明

### 7.1 OEM

**用途**：监控事实来源  
**技术说明**：
- 承载 Metric Extension
- 采集监控值
- 保存历史趋势
- 生成告警与事件
- 通过 OMS 官方接口供外部系统读取

**边界**：
- 不负责自然语言交互
- 不建议通过 repository 数据库直连方式做取数

### 7.2 Grafana

**用途**：图表展示与趋势钻取  
**技术说明**：
- 通过 Oracle 官方 EM App for Grafana 与 OEM 集成
- 展示趋势曲线、横向对比、多指标联动
- 可作为 AI 回答后的图表承接层

**边界**：
- 不负责核心推理
- 不负责 SOP 解释

### 7.3 AI Gateway

**用途**：问题理解与查询编排  
**技术说明**：
- 做意图识别
- 做对象识别
- 调用 OEM 接口
- 调用知识库
- 生成统一风格回答
- 输出 Grafana 链接

**边界**：
- 第一阶段建议只做只读问答
- 不建议一开始就做自动修复或自动执行

### 7.4 知识库

**用途**：文档和经验沉淀  
**技术说明**：
- 可基于 Oracle 26ai 或企业现有 RAG
- 承载 SOP、经验、官方文档、历史案例
- 为 AI Gateway 提供补充说明和建议内容

**边界**：
- 需要持续维护
- 内容应可追溯、可更新

### 7.5 客户端

**用途**：用户统一入口  
**技术说明**：
- 第一阶段优先用 CLI
- 后续可扩展到 Web
- 只负责提问与展示结果

**边界**：
- 不应把复杂配置工作压给终端用户

## 8. 为什么这套组合合理

如果只做 AI，而没有 OEM、Grafana 和知识库，往往会出现几个问题：

- 没有 OEM，AI 没有可靠实时数据来源
- 没有 Grafana，用户无法快速看趋势和 drill-down
- 没有知识库，AI 只能“报数”，不能“解释”
- 没有 AI Gateway，这些组件之间很难形成顺畅的统一入口

而本方案恰好把各自最擅长的部分拼在一起：

- OEM 负责“采”和“告”
- Grafana 负责“看”
- AI Gateway 负责“问”和“串”
- 知识库负责“解释”和“SOP”
- 客户端负责“让人用起来简单”

这比单纯堆一个大模型，更符合实际运维场景，也更容易长期维护。

## 9. 建议的落地路径

### 阶段 1：试点上线

建议先聚焦这两个一体机硬件指标：

- Memory HardwareCorrupted
- DiskErrorCount

先实现：

- 只读问答
- 当前异常解释
- 趋势变化判断
- 风险优先级提示
- SOP 建议
- Grafana 链接跳转

目标是：**先让用户愿意用、觉得有帮助。**

### 阶段 2：场景扩展

当这两个指标试点稳定后，可以逐步扩展到：

- 更多主机硬件项
- Exadata 其他指标
- 数据库性能指标
- 主机层 CPU / I/O / 内存 / 网络联动分析

### 阶段 3：自动化衔接

在问答稳定后，再考虑和自动化流程做衔接，例如：

- 生成标准化处理建议
- 对接 Ansible
- 做半自动巡检或排查辅助

原则上建议：

- **先读、后写**
- **先建议、后执行**

## 10. 汇报结论

这套方案的核心价值在于：

1. **不改 OEM 本体**
2. **不碰 OEM repository 数据库账号密码**
3. **复用 OEM 现有监控能力和官方接口**
4. **把复杂的监控数据变成简单的问答入口**
5. **让用户在“问、看、处理”之间形成顺畅闭环**

对于当前这两个一体机指标来说，这套方案已经具备试点条件。因为数据、脚本、阈值和告警基础都在 OEM 中，真正缺的是一个更适合运维人员使用的智能入口。

用一句话总结就是：

**OEM 负责把数据采上来，Grafana 负责把图看清楚，知识库负责把经验讲明白，AI Gateway 负责把这些东西串成一个真正能用的运维入口。**
