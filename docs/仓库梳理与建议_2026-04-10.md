# askoem_mcp_skill 仓库梳理与建议（2026-04-10）

## 1. 我对当前仓库的新理解

### 1.1 系统定位
该仓库本质是一个 **“OEM 只读诊断 + MCP Tool + LLM Skill” 的分层式 MVP**：
- 数据层通过 MCP tools 对 OEM 做只读取数（登录、指标、事件、告警）。
- 智能层把 `skills/*/SKILL.md` 作为可扩展的“认知插件”，由 LLM 做技能路由与执行。
- 对外输出固定结构化诊断结果（结论/证据/SOP/下一步），并且保留降级路径。

### 1.2 主链路（我按代码重新抽象）
1. `run_skill` / `fetch_data_from_oem` 接收问题与认证参数。
2. `intent_parser` 做意图识别、目标提取、时间范围识别、route_key 映射。
3. `alert_router` 先规则判定，再低置信度触发 LLM fallback 分类。
4. `oem_client` 拉取 latestData / metricTimeSeries / incidents / events。
5. `skill_engine` 扫描 Skill，LLM 路由后按完整 SKILL.md 执行。
6. 若 Skill 未命中或 LLM 不可用，退化为数据摘要文本。

### 1.3 当前设计亮点
- **强约束分层清晰**：数据层与推理层职责分离，便于治理。
- **Skill 扩展低成本**：新增场景主要写 SKILL.md，不必改主代码。
- **兼容真实环境差异**：有 session 缓存、HTTP/SSE 启动入口、以及低版本 OEM 兼容策略。
- **可演示可落地**：MVP 对“问答-取数-诊断-返回”闭环已经完整。

---

## 2. 主要风险与问题（按优先级）

### P0：配置与安全风险
1. `config/metric_map.yaml` 中存在默认内网地址，且 `.env` 文件出现在仓库根目录；建议彻底改为模板化与环境注入，避免真实环境信息误提交。
2. 当前 README 提到 `.env.example`，但仓库中未看到该文件，存在“文档与实际不一致”。

### P1：可靠性与可维护性风险
1. `intent_parser` 的规则策略较硬编码，存在拼写异常（如 `database instsance`）与中英文语义覆盖不足，后续规模化会出现误判。
2. `oem_client.list_recent_incidents` 当前是模拟数据兜底，虽然符合“先跑通”，但若未加显式开关会导致生产语义偏差。
3. `skill_engine` 使用正则解析 YAML frontmatter，复杂 YAML（多行列表、引号、注释嵌套）下鲁棒性不足。
4. `mcp_server_http` 写死 SSE 路由风格，文档里提到 streamable-http，但当前实现侧重 SSE，存在“可选能力描述 > 实际能力”的风险。

### P2：工程化不足
1. 自动化测试、静态检查、CI 流水线信息不明显（仓库内未见测试目录）。
2. 可观测性不足：缺少统一日志 trace_id、耗时指标、错误分层码。
3. Skill 生命周期治理尚弱：版本、审批、回滚、灰度规则未形成体系。

---

## 3. 建议路线图（可分三期）

### 第一期（1-2 周，先补“生产底座”）
- 配置治理：
  - 增加 `.env.example` 与 `config/metric_map.example.yaml`。
  - 清理仓库中的真实/半真实默认地址。
- 安全治理：
  - 增加敏感信息扫描（pre-commit + CI secret scan）。
  - 明确凭据来源优先级（参数 > 环境变量 > secret manager）。
- 基础质量：
  - 补充最小测试：intent 解析、路由分类、skill 注册、fallback 行为。

### 第二期（2-4 周，增强“准确性与一致性”）
- 意图识别升级为“规则 + 轻量分类模型”双轨，提升多语言混输场景精度。
- incidents mock 改为显式 feature flag（如 `OEM_INCIDENTS_MOCK=true`），并在返回体标明数据来源等级。
- Skill 元数据解析改用 `yaml.safe_load`，正则仅作兜底。
- 输出协议版本化（例如 `result_schema_version`），方便客户端长期兼容。

### 第三期（4-8 周，面向规模化）
- Skill 管理中心化：版本审批、A/B 路由、灰度发布、回滚。
- 引入评测集（黄金问答 + 人工标注）做离线回归评估。
- 建立链路观测：请求量、成功率、Skill 命中率、LLM 调用耗时、OEM 失败分布。
- 逐步加入“建议级自动化”（只生成执行计划，不直接执行高风险动作）。

## 5. 结论
这个仓库已经具备很好的 MVP 骨架，尤其在“分层明确、Skill 可扩展、MCP 兼容部署”方面方向正确。下一阶段不应急于堆新功能，而应优先补齐 **配置安全、测试覆盖、解析鲁棒性、可观测性** 四个基础能力。这样后续无论新增 IO/HBA/数据库等 Skill，还是接入更多客户端（VS Code/Cline/Web），都能保持稳定演进。
