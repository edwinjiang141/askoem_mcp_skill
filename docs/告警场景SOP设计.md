# 日常告警场景设计（OEM incidents/events 主路径）

## 1. 问题识别方案（混合模式）

在 Cline 中已配置 DeepSeek 的前提下，AI Gateway 采用：

1. **规则优先**（默认）
   - 关键词 + 语义短语匹配，直接识别三类场景：
     - `cpu_high`
     - `io_high`
     - `hardware_hba_disk`
2. **LLM 回退**（仅低置信度）
   - 当规则识别置信度 `< 0.8` 时，调用可选 LLM 分类器。
   - LLM 只做“标签分类”，不做处置决策，避免幻想影响流程稳定性。

> 结论：识别层是“规则控制”，LLM 只是补充兜底。

## 2. 告警数据来源

主来源统一为：

- `OEM incidents`
- `OEM incident events`

当前版本不读取补充指标，仅基于告警对象生成处置建议。
在低版本 OEM 环境下，`incidents` 会按提问内容（目标名/场景/时间提示）构造模拟样本，确保 SOP 链路可运行。

## 3. SOP 执行模型

### 3.1 CPU 高告警 SOP（已实现）

执行步骤：

1. 拉取目标最近 24 小时 incidents/events。
2. 依据固定步骤给出处置建议：
   - 是否持续高位
   - top 进程/SQL 定位
   - 与 IO 等待联动校验
   - 临时缓解与升级策略
   - 阈值复盘建议

### 3.2 IO 逻辑读/物理读高 SOP（已实现）

执行步骤：

1. 拉取目标最近 24 小时 incidents/events。
2. 依据固定步骤给出处置建议：
   - 区分逻辑读 vs 物理读
   - Top SQL / 执行计划校验
   - 存储延迟与队列深度排查
   - 变更窗口关联
   - SQL 治理与容量评估

### 3.3 HBA / Disk 硬件告警（预留）

当前保留接口与输出模板，待你提供硬件场景 SOP 细节后可直接接入。

## 4. 无专用 SOP 时的通用流程

通用 SOP 固定为：

1. 告警确认（严重级别、持续时长、影响范围）
2. 证据关联（同时间窗口指标和事件）
3. 根因定位（计算/存储/网络/硬件）
4. 临时缓解（降载、隔离、限流）
5. 永久修复（配置优化、容量治理、流程改进）
6. 复盘闭环（阈值与告警策略优化）

## 5. 实现边界

- 不依赖知识库判断故障根因。
- 不把决策交给 LLM。
- 不执行自动修复动作，仅提供建议。

## 6. 可维护性（本次合并优化）

- 告警识别规则已从代码常量下沉到 `config/metric_map.yaml.alert_scenarios`。
- 每个场景可配置：
  - `keywords`
  - `require_target`
- 新增场景时优先改配置，不建议复制/粘贴新的流程代码。
