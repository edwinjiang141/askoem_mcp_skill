# AI Gateway 说明

## 1. AI Gateway 是什么

AI Gateway 可以理解成一个**运维问答中间层**。

它不是 OEM，也不是 Grafana，更不是一个替代现有运维平台的新系统。  
它的作用是站在 **终端用户** 和 **OEM / Grafana / 知识库 / 大模型** 之间，把原本分散的能力串起来：

- 从 OEM 取监控数据、事件、告警
- 从知识库取文档、SOP、经验
- 把这些信息整理后交给大模型
- 输出成运维人员能直接看懂、能直接用的回答

换句话说，AI Gateway 的核心价值不是“自己生产数据”，而是**把已有数据变成可问、可解释、可行动的信息**。

## 2. 为什么需要 AI Gateway

如果没有 AI Gateway，现有几套能力其实是割裂的：

### OEM 的问题
OEM 很强，能采集、能告警、能看历史趋势，但它更像“监控平台”，不是“问答入口”。  
运维人员想快速问一句“这台主机为什么异常”，OEM 本身并不擅长直接回答这种问题。

### Grafana 的问题
Grafana 很适合画图、看趋势、做 dashboard，但它不负责解释“为什么出问题”，也不负责结合 SOP 给建议。

### 知识库的问题
知识库能回答“怎么处理”，但通常不知道“当前系统到底发生了什么”。

### 大模型的问题
大模型很会总结，但如果没有受控的数据输入，就容易变成泛泛而谈，甚至答非所问。

所以 AI Gateway 的意义就在于：  
**让 OEM、Grafana、知识库、大模型形成一个完整流程，而不是各自孤立。**

## 3. AI Gateway 的定位

AI Gateway 最适合被定义成：

**一个只读优先、面向运维问答的编排层。**

这里有几个关键词很重要：

### 只读优先
第一阶段只做：

- 查状态
- 查指标
- 查趋势
- 查告警
- 查 SOP
- 给建议

不做：

- 自动改配置
- 自动执行修复
- 自动调用高风险动作
- 自动变更 OEM 配置

因为先把“看得懂、说得清、建议准”做好，比一上来做自动执行更实际。

### 编排层
它不应该自己重造一套监控系统，而是调用现有系统：

- OEM 提供 Telemetry
- Grafana 提供可视化
- 知识库提供 Documentation / SOP
- LLM 提供语言组织和解释能力

### 运维问答入口
终端用户不需要懂 REST API、metric group、事件模型、Grafana 变量这些实现细节。  
用户只需要像问同事一样提问。

## 4. AI Gateway 的核心职责

前面讨论下来，AI Gateway 最核心的职责一共 5 个：

### 1）接收用户问题
比如：

- 哪些数据库告警最严重？
- 某台主机为什么异常？
- Exadata 最近有什么硬件风险？
- 这个指标高了意味着什么？
- 遇到这个告警应该先查什么？

### 2）识别问题类型
把问题归到固定几类，而不是让模型完全自由发挥。比如：

- 当前状态查询
- 告警汇总
- 单目标异常诊断
- 趋势变化解释
- 文档 / SOP 问答
- 风险排序

### 3）去对应系统取数据
不同问题走不同数据源：

- OEM：监控指标、目标属性、事件、告警
- 知识库：排查步骤、产品说明、内部 SOP
- Grafana：作为后续 drill-down 的可视化落点

### 4）组织成结构化上下文
把原始数据整理成模型能稳定理解的格式，比如：

- 目标信息
- 当前指标值
- 最近 1 小时 / 24 小时趋势摘要
- 相关 incident / event
- 相关 SOP 片段

### 5）生成最终回答
输出不是一大段空话，而是固定结构：

- 结论
- 证据
- 影响范围
- 下一步建议
- 可点击的 Grafana 链接

## 5. AI Gateway 不应该做什么

这一点也很重要。

AI Gateway 第一阶段不建议做下面这些事：

### 1）不直接访问 OEM repository 数据库
这是前面反复讨论过的硬限制。  
客户明确不允许通过用户名/密码访问 OEM 数据库，所以方案必须彻底绕开这条路。

### 2）不替代 OEM
它不是新监控平台，不负责采集，不负责阈值配置，不负责告警生成。

### 3）不替代 Grafana
它不负责画复杂图表，不负责多维 dashboard 展示。

### 4）不做完全自由式 Agent
不要让 AI 自己去随意决定访问什么接口、查哪些数据、执行什么动作。  
第一阶段最稳的方式还是**固定模板 + 固定流程 + 固定输出**。

### 5）不直接执行高风险动作
比如自动 blackout、自动执行修复、自动改配置，这些都不适合第一版。

## 6. AI Gateway 的推荐架构

前面讨论后，一个比较稳的架构可以概括成下面这样：

### 用户侧

- CLI
- 或后续可补一个轻量 Web 页面

### AI Gateway
内部可分成几个逻辑模块：

#### A. API / Session 层
负责接收请求、维持会话、记录审计。

#### B. Intent Router
负责判断问题类型和目标对象。

#### C. OEM Connector
负责访问 OMS，读取：

- targets
- properties
- metric groups
- latestData
- metricTimeSeries
- incidents
- events

#### D. Knowledge Connector
负责从知识库中取：

- Oracle 文档
- 内部 SOP
- 历史经验
- 常见故障说明

#### E. Answer Composer
把 OEM 数据和知识库内容拼装成标准化上下文，再交给 LLM 输出。

#### F. Link Resolver
根据问题场景选定对应 Grafana dashboard / panel 链接，并带上时间范围和目标名。

## 7. AI Gateway 如何取数

这是整套方案里最关键的一点。

前面已经确定，AI Gateway 的取数原则是：

## **优先走 OEM REST API**
这是主路径。

适合做：

- 查目标
- 查属性
- 查 metric groups
- 查最新指标值
- 查时间序列
- 查 incidents / events

这样做的好处是：

- 官方支持
- 结构清晰
- 更适合程序化调用
- 不碰 OEM 数据库
- 比直接让终端用户跑脚本更可控

## **EM CLI 作为补充**
EM CLI 不建议做主取数路径，但可以作为补充层：

- 某些 REST 不方便的管理动作
- 某些补充脚本化能力
- 后续与自动化流程衔接

也就是说：

**CLI 是用户入口的一种形式，但后端主取数层最好仍然是 REST API。**

## 8. AI Gateway 如何和 Grafana 配合

Grafana 在整个方案里不是 AI 本体，而是 AI 的“深挖入口”。

更准确地说：

- AI Gateway 负责理解问题、组织答案
- Grafana 负责让用户继续看图、看趋势、看 drill-down

所以二者的关系是：

### AI Gateway 负责“说清楚”
比如：

- 当前有哪些异常
- 这个异常值是否超过阈值
- 最近是在上升还是持续高位
- 下一步应该先看什么

### Grafana 负责“看清楚”
比如：

- 过去 1 小时趋势图
- 多主机横向对比
- 单主机多指标联动
- 单磁盘 / 单 cell 维度细查

所以最自然的做法是：  
**AI 回答里附 Grafana dashboard / panel 链接。**

用户如果只是想快速知道结论，看 AI 回答就够了；  
如果想继续深挖，就点 Grafana 链接。

## 9. AI Gateway 如何和知识库配合

前面讨论里，Ask EM 的核心是两块：

- Telemetry
- Documentation

AI Gateway 就是在本地把这两块重新拼起来：

### Telemetry
来自 OEM：

- 监控数据
- 事件
- 告警
- 趋势

### Documentation
来自知识库：

- Oracle 官方文档
- 企业内部 SOP
- 一体机运维经验
- 典型故障案例

这样 AI 才能做到：

不是只说“这个值高了”，  
而是还能继续说“这通常意味着什么、下一步怎么查”。

## 10. AI Gateway 的部署建议

最实用的第一版部署方式是：

### 一台内部 Linux VM 或容器主机
上面部署：

- Gateway 服务
- 缓存
- 本地检索组件
- 与内网模型服务的连接

### 网络要求
Gateway 需要能访问：

- OMS HTTPS 地址
- Grafana 服务
- 知识库 / 向量检索服务
- 本地 LLM 服务

### 为什么推荐先做单实例
因为第一阶段重点不是高可用，而是验证：

- 回答是否准确
- 取数是否稳定
- 用户是否真的愿意用

验证通过后，再考虑双实例、负载均衡、共享缓存这些高可用设计。

## 11. AI Gateway 的认证与权限思路

这块前面也反复讨论过，建议分阶段做。

### 第一阶段：最稳的做法

- 用户登录 AI Gateway
- AI Gateway 后端使用一个专用的 OEM 只读账号访问 OMS
- 这个账号只给最小必要权限
- 不用 SYSMAN，不用 Super Admin

这样实施简单、风险可控。

### 第二阶段：更细粒度做法
如果客户要求更严格，可以再做成：

- 用户通过企业统一认证登录 Gateway
- Gateway 根据用户身份映射 OEM 权限范围
- 不同用户只能看到自己负责的 targets

但这一版复杂度明显更高，不建议作为第一步。

## 12. AI Gateway 的典型交互流程

可以概括成下面这条链路：

### 第一步：用户提问
例如：

- x9mdbadm01 这台主机为什么有硬件内存告警？
- x9mceladm01 最近有没有磁盘错误上升？

### 第二步：Gateway 判断意图
比如：

- 是单目标诊断
- 还是趋势分析
- 还是处理建议

### 第三步：Gateway 去 OEM 取数
包括：

- target
- metric latest value
- time series
- incidents / events

### 第四步：Gateway 去知识库补内容
包括：

- 该指标含义
- 排查 SOP
- 经验说明

### 第五步：Gateway 选 Grafana 链接
比如：

- 主机硬件 dashboard
- Exadata 磁盘错误 dashboard

### 第六步：Gateway 生成回答
按固定结构返回：

- 结论
- 证据
- 建议
- 可继续深挖的链接

## 13. AI Gateway 适合承载哪些场景

最适合先做的，是这些“高频、低风险、价值明显”的场景：

### 1）当前异常问答

- 现在哪些目标最危险
- 哪台主机有硬件风险
- 哪个数据库当前最值得先处理

### 2）单目标解释

- 这台主机为什么报警
- 这个数据库为什么变慢
- 这个 Exadata 有什么异常

### 3）趋势解释

- 最近 1 小时有没有明显变坏
- 最近 24 小时哪些盘错误计数在上升

### 4）SOP / 文档问答

- 这个告警一般先看什么
- 这个指标高意味着什么
- 有哪些标准排查步骤

## 14. AI Gateway 的输出应该长什么样

这一点很重要。

我建议固定成下面这个结构：

### 结论
直接说判断结果，不绕弯。

### 证据
给出：

- 目标名
- 指标名
- 当前值
- 阈值
- 时间范围
- 相关告警 / 事件

### 下一步建议
给 2~3 条最实际的动作。

### 深挖入口
附 Grafana 链接，必要时附文档 / SOP 入口。

这种格式对运维人员最友好，因为它既不像原始监控界面那样零散，也不像普通 AI 那样只会说一段大白话。

## 15. AI Gateway 的价值总结

把前面讨论压缩成一句话：

**AI Gateway 的价值，不是取代 OEM 或 Grafana，而是把 OEM 的监控能力、Grafana 的可视化能力、知识库的文档能力和大模型的总结能力整合成一个真正可用的运维问答入口。**

更直白一点：

- OEM 负责“有数据”
- Grafana 负责“看得清”
- 知识库负责“有经验”
- 大模型负责“会表达”
- AI Gateway 负责“把这些拼成一个完整流程”

## 16. 一个适合内部汇报的简短结论

你可以这样概括 AI Gateway：

**AI Gateway 是一个面向运维场景的中间层。它不直接采集数据，也不替代 OEM 和 Grafana，而是通过 OEM REST API 读取监控数据和告警信息，通过知识库补充文档和 SOP，再结合大模型把结果整理成“结论 + 证据 + 下一步建议”的形式返回给用户。对于用户来说，入口可以只是一个简单的 CLI 或轻量页面；对于平台来说，它则是把 Telemetry、Documentation 和可视化联动起来的核心编排层。**
